### 1. Настройка Keycloak (PKCE и Audience Mapper)

1. **Реализация PKCE (Proof Key for Code Exchange)**
   - В конфигурационном файле `realm-export.json` добавлен атрибут `"pkce.code.challenge.method": "S256"`.
   - Используется метод `S256` благодаря хэшированию (SHA-256), что обеспечивает необратимость вычисления `code_challenge` и повышенную безопасность по сравнению с методом `plain`.

2. **Настройка Audience Mapper**  
   Чтобы токен, выданный клиенту `reports-frontend`, содержал `Audience` для `reports-api` и корректно проходил валидацию на стороне бэкенда, в конфигурации клиента в `realm-export.json` добавлен соответствующий маппер:
   ```json
   "protocolMappers": [
     {
       "name": "audience-reports-api",
       "protocol": "openid-connect",
       "protocolMapper": "oidc-audience-mapper",
       "consentRequired": false,
       "config": {
         "included.client.audience": "reports-api",
         "id.token.claim": "false",
         "access.token.claim": "true"
       }
     }
   ]
   ```
   Данный маппер гарантирует, что при авторизации будет выдаваться токен с нужными Audience для корректной работы с сервисом `reports-api`. если воспользовать сервисом jwt.io для декодирования сформированного токена мы увидимо поле `"aud": "reports-api"`

3. **Интеграция с фронтендом**
   - При использовании библиотеки `@react-keycloak/web` вместе с `keycloak-js` в опциях инициализации добавлен параметр `pkceMethod: 'S256'`.
   - Таким образом, при авторизационном запросе токен защищен от атак типа «перехват кода авторизации».

### 2. Создание backend-части приложения для API

В рамках задачи по реализации сервиса, который будет выдавать отчёты, создано отдельное backend-приложение на языке C# (.NET).
- Реализован **эндпоинт** `GET /reports`, который на данном этапе возвращает **фиктивные данные** (без подключения к реальным БД).
- Предполагается, что при последующей доработке приложение будет интегрировано с ClickHouse и CRM для получения реальных пользовательских данных и формирования отчётов.

---

### 3. Инструкция по запуску и проверке

Для демонстрации работы приложения используется Docker.

1. **Запуск приложения**
   ```bash
   docker compose up -d --build
   ```
   В результате контейнеры будут собраны и запущены в фоне.

2. **Авторизация**
   - Откройте в браузере frontend-приложение `http://localhost:3000/`
   - Перейдите на страницу авторизации (нажав кнопку `Login`) и войдите под одним из пользователей. Например:
      - **user**: `prothetic1`
      - **pass**: `prothetic123`
   - После успешной авторизации можно обращаться к эндпоинту `/reports` (нажав кнопку `Download Report`). При наличии соответствующих прав будет возвращён отчёт (фиктивные данные).

3. **Проверка прав**
   - Чтобы проверить разницу в доступах, выполните `Logout` и зайдите под другим пользователем:
      - **user**: `admin1`
      - **pass**: `admin123`
   - Для данного пользователя доступ к `/reports` будет запрещён, и сервис вернёт соответствующее сообщение об ошибке прав доступа.

1. В **Keycloak** настроены механизмы PKCE и добавлен **Audience Mapper** для корректной валидации и выдачи токенов.
2. Реализован **минимальный backend** на C# (.NET) с эндпоинтом `/reports`, возвращающим тестовые данные, а также подготовлена инструкция по запуску и проверке различий в правах доступа.
